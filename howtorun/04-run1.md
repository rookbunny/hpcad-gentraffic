# How To Run: Run1 (Opportunistic Intruder)

## Purpose

- Capture the benign scaffold underneath a loud, opportunistic attack.
- Window is 20 minutes (1200 seconds), which is 80 buckets at a 15 second span.
- The only systematic difference from the baseline is the attack itself, so the benign substrate is generated the same way and every attacker action is logged as it happens.

## Processes in this profile

- `chat_ws`
- `healthcheck`
- `email`
- `prom_scrape` (runs on the monitoring VM)

Note that `run1` omits the `noaa` space-weather pulls that appear in `baseline` and `run2`. The window is short and the attack is loud, so the long external transfer is left out.

## What is not generated here

- The C2 beacon is hand-run in the C2 framework, not by this generator.
- Manual actions (brute force, privilege escalation, C2 interaction) are performed by the operator and recorded with `log_user.py`.

## Steps

1. On the capture host, start the run with a fresh seed. The script prints the run id, the tag, and the seed.

   ```bash
   sudo ./run_capture.sh run1 ens19
   ```

   `run_capture.sh` writes the active tag to `.current_run`, so `log_user.py` picks it up automatically for the rest of the run.

2. On the monitoring VM, start the scrape process with the same seed and run id.

   ```bash
   ./.venv/bin/python3 gen.py --profile run1 --role promsvc \
       --seed <seed> --run-id <run_id>
   ```

3. As the opportunistic attack proceeds, record each attacker action the moment it happens. The tag is read from `.current_run`, so it does not need to be passed. Add an ATT&CK id with `--phase` where one applies.

   ```bash
   ./.venv/bin/python3 log_user.py "hydra ssh brute force start" --source attacker --phase T1110
   ./.venv/bin/python3 log_user.py "valid ssh login gained"       --source attacker --phase T1078
   ./.venv/bin/python3 log_user.py "sliver interactive shell open" --source attacker
   ```

4. Let the window run out. The driver exits at 20 minutes, stops tcpdump, and finalizes the manifest.

## Reseeding

- `run1` uses a seed distinct from `baseline` and `run2`. To reproduce an earlier `run1`, pin its seed.

  ```bash
  sudo ./run_capture.sh run1 ens19 24680
  ```

## Outputs

Under `logs/<tag>_logs/`, where the tag is `R<run_id>-S<seed>`:

- `<tag>.pcap`
- `<tag>_knownbenign.json` (scripted benign events)
- `<tag>_knownuser.json` (attacker actions logged during the run)
- `<tag>_alltraffic.json` (union of both, keyed by the `class` field)
- `<tag>.<role>.manifest.json`

Within the run window only the scripted benign set, the recorded manual actions, and the emulated adversary are present. Any packet not attributable to a benign or user record belongs to the adversary.

## Verification checklist

- The manifest reports `expected_buckets` of 80.
- Every attacker action taken during the window has a matching record in `<tag>_knownuser.json` with `class=user`.
- The capture host and the monitoring VM share the same tag.
- The bucket span is 15 seconds, matching the baseline that this run is scored against.
