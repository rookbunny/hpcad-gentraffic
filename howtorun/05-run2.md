# How To Run: Run2 (Covert Long-Dwell Operator)

## Purpose

- Capture the benign scaffold underneath a covert, low-and-slow adversary.
- Window is 40 minutes (2400 seconds), which is 160 buckets at a 15 second span.
- Live web browsing is run during this window as cover, so the covert C2 rhythm has realistic human traffic to hide inside.

## Processes in this profile

- `chat_ws`
- `healthcheck`
- `email`
- `noaa` (external space-weather archive pulls)
- `prom_scrape` (runs on the monitoring VM)

## What is not generated here

- The C2 beacon is hand-run in the C2 framework at its covert cadence, not by this generator.
- Manual web browsing and every attacker action are performed by the operator and recorded with `log_user.py`.

## Per-destination periodicity reminder

- The beacon, an enabled Zabbix stream, and the healthcheck all share a 60 second cadence by design.
- Detection features must group flows by responder (destination IP, or IP plus port) before computing inter-arrival and periodicity, so the unit of analysis is `(destination, window)` rather than `window`. Aggregated per bucket, the three 60 second streams sum into one signal and the beacon hides. Computed per destination, the beacon surfaces as its own periodic endpoint.

## Steps

1. On the capture host, start the run with a fresh seed. The script prints the run id, the tag, and the seed, and writes the active tag to `.current_run`.

   ```bash
   sudo ./run_capture.sh run2 ens19
   ```

2. On the monitoring VM, start the scrape process with the same seed and run id.

   ```bash
   ./.venv/bin/python3 gen.py --profile run2 --role promsvc \
       --seed <seed> --run-id <run_id>
   ```

3. Run live browsing as cover during the window. Route an operator browser through an SSH dynamic proxy into the honeypot so the browsing egresses from the honeypot uplink and reaches the mirror. See the testing document for the tunnel setup.

   ```bash
   ssh -D 1080 -N operator@<honeypot_ip>
   ./.venv/bin/python3 log_user.py "youtube session start" --source browsing
   # browse for several minutes through the proxied browser
   ./.venv/bin/python3 log_user.py "youtube session stop"  --source browsing
   ```

4. Record each covert attacker action as it happens. The tag is read from `.current_run`. Add an ATT&CK id with `--phase` where one applies.

   ```bash
   ./.venv/bin/python3 log_user.py "beacon check-in observed"      --source attacker
   ./.venv/bin/python3 log_user.py "credential access via service" --source attacker --phase T1078
   ```

5. Let the window run out. The driver exits at 40 minutes, stops tcpdump, and finalizes the manifest.

## Reseeding

- `run2` uses a seed distinct from `baseline` and `run1`. To reproduce an earlier `run2`, pin its seed.

  ```bash
  sudo ./run_capture.sh run2 ens19 13579
  ```

## Outputs

Under `logs/<tag>_logs/`, where the tag is `R<run_id>-S<seed>`:

- `<tag>.pcap`
- `<tag>_knownbenign.json` (scripted benign events)
- `<tag>_knownuser.json` (browsing and attacker actions logged during the run)
- `<tag>_alltraffic.json` (union of both, keyed by the `class` field)
- `<tag>.<role>.manifest.json`

## Verification checklist

- The manifest reports `expected_buckets` of 160.
- Browsing sessions and covert attacker actions all appear in `<tag>_knownuser.json` with `class=user`.
- The proxied browsing shows up on the mirror as flows sourced from the honeypot.
- The capture host and the monitoring VM share the same tag.
- The bucket span is 15 seconds, matching the baseline that this run is scored against.
